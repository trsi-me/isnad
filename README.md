## 1. نبذة عن المشروع

**إسناد** تطبيق جوال مبني بإطار عمل **Flutter** يهدف إلى **محاكاة** نظام إدارة بلاغات عسكرية ميدانية ضمن نطاق مشروع تخرّج أو عرض تقني: يسجّل العسكري البلاغ (نوع الإصابة، الموقع، الوصف، صورة اختيارية)، وتتابع **القيادة** أو **المستشفى** مسار الحالة من مرحلة «مفتوح» حتى «مغلق»، مع إمكانية إدخال **محضر إصابة** عند إغلاق البلاغ من قبل المستشفى بعد وصول الفريق ميدانياً.

من الناحية البرمجية، التطبيق يطبّق **فصلاً بين الطبقات**: واجهات (`screens` + `widgets`)، منطق حالة (`providers`)، وصول للبيانات (`database_helper` + نماذج)، وخدمات جانبية (`services`) — وهذا **يسهّل** إضافة خادم لاحقاً دون إعادة كتابة الواجهة بالكامل.

### لمن هذا المشروع؟

- طالب يريد مشروعاً **يعمل على الهاتف فوراً** دون إعداد خادم.
- فريق يريد **واجهة عربية RTL** وهوية بصرية موحّدة (ألوان نيلية/ذهبية/فضية).
- من يريد قاعدة انطلاق لاحقاً لربط **API** حقيقي أو **Firebase** كاملاً.

### ما الذي لا يفترضه الإصدار الحالي؟

- لا يوجد **خادم مركزي** حقيقي؛ المزامنة **وهمية** (تحديث `is_synced` في SQLite).
- كلمات المرور التجريبية **غير مشفّرة** في قاعدة البيانات المحلية (للعرض فقط).
- إشعارات **Firebase Cloud Messaging** تحتاج إعداد مشروع Firebase وملفات المنصة؛ التطبيق يعمل بدونها مع إشعارات محلية حيث ينطبق ذلك.

المشروع مصمّم ليكون **جاهزاً للتشغيل محلياً** باستخدام **SQLite** على الجهاز، مع **مزامنة وهمية** عند توفر الاتصال.

## 2. الميزات الرئيسية

### للعسكري الميداني

- تسجيل دخول بحساب تجريبي وعرض **الرئيسية** ببطاقة ترحيب واختصارات واضحة.
- **بلاغ جديد**: اختيار نوع الإصابة (شبكة 2×3)، سحب الموقع، وصف اختياري، إرفاق صورة من المعرض (حسب `CameraService.pickImage`).
- **بلاغاتي**: قائمة بلاغات المستخدم مع شارة حالة؛ فتح **تفاصيل** مع **خط زمني** للمراحل.
- **محاضر الإصابة**: عرض سجلات مرتبطة بحسابه لبلاغات **مغلقة** وفق الاستعلام في المزوّد.
- **SOS**: زر عائم أحمر ينشئ بلاغ طوارئ بسرعة مع إحداثيات (إن وُجدت) وإشعار محلي.
- **الملف الشخصي** و**الإشعارات** (شاشة إشعارات توضيحية).

### للقيادة وللمستشفى (نفس الشاشة الرئيسية مع اختلاف الصلاحيات)

- تبويب **البلاغات النشطة** (حالة ليست `closed`) وتبويب **الكل**.
- زر **خريطة مباشرة** مع بلاطات OpenStreetMap وعلامات للبلاغات ذات إحداثيات.
- **تفاصيل البلاغ**: عرض بيانات كاملة؛ أزرار استجابة حسب الحالة الحالية فقط.
- **المستشفى**: عند **إغلاق الحالة** من مرحلة «تم الوصول» يظهر حوار لإدخال بيانات تقرير طبي تُحفظ في `injury_records`.

### عام

- جلسة محفوظة عبر `SharedPreferences` بعد تسجيل الدخول.
- اتجاه واجهة **RTL** ولغة `ar` في `MaterialApp`.
- خط **IBM Plex Sans Arabic** فقط (موجود تحت `assets/fonts/`).

## 7. قاعدة البيانات

**SQLite**: قاعدة بيانات تُخزَّن كملف واحد على الجهاز دون خادم. مثال واقعي: دفتر بيانات داخل الهاتف يبقى يعمل أثناء انقطاع الشبكة. في إسناد: نخزّن المستخدمين والبلاغات والاستجابات والمحاضر، ونستخدم `is_synced` لمحاكاة «إرسال» البلاغات عند عودة الاتصال.

الملف يُنشأ عبر `openDatabase` في مسار يعادل `getDatabasesPath()` مع اسم مثل `isnad.db`. يُفعّل `PRAGMA foreign_keys = ON` عند الإعداد لضمان اتساق الإدراجات مع المفاتيح الأجنبية (على مستوى SQLite).

### الجداول والعلاقات (بتفصيل الحقول المنطقي)

- **`users`**: المفتاح `id`؛ `military_id` فريد؛ `role` بقيم نصية (`soldier` / `command` / `hospital`) تطابق `UserRole.value`؛ `password_hash` في الوضع التجريبي يخزّن كلمة المرور كنص للمقارنة المباشرة — **لا يُقلّد في الإنتاج**.
- **`reports`**: `report_uuid` معرف عرض وطابع زمني؛ `military_id` + `reporter_name` لعرض المُبلّغ؛ `injury_type` يطابق اسم الـ enum (`gunshot`…`other`)؛ `status` يطابق `ReportStatus`؛ `is_sos` و`is_synced` كأعداد صحيحة 0/1 في SQLite تُحوَّل إلى `bool` في النموذج؛ `created_at` / `updated_at` كنصوص ISO8601 للترتيب والعرض.
- **`responses`**: يربط `report_id` بـ `reports.id`؛ `action_type` يصف الإجراء (`received`, `team_sent`, `arrived`, `closed`)؛ `action_time` لترتيب سجل الاستجابات زمنياً في الواجهة.
- **`injury_records`**: `report_id` فريد (محضر واحد لكل بلاغ)؛ `military_id` يربط المحضر بالجندي؛ `legal_status` افتراضياً `pending` مع إمكانية التوسع لاحقاً.

### البذر (Seed)

عند أول تشغيل بعد إنشاء الملف، إذا كان جدول `users` فارغاً، تُدرج الحسابات التجريبية الثلاثة تلقائياً من `DatabaseHelper.ensureSeeded()`.

### فهرسة وترتيب

الاستعلامات الشائعة تستخدم `ORDER BY created_at DESC` للبلاغات لعرض الأحدث أولاً؛ محاضر الإصابة تُربَط بـ `JOIN` بين `injury_records` و`reports` لتصفية البلاغات المغلقة فقط.

## 10. شرح مفصل للكود مع مقتطفات

### أ) نقطة الدخول `main.dart`

```dart
void main() async {
  WidgetsFlutterBinding.ensureInitialized();
  await initializeDateFormatting('ar');
  try {
    await Firebase.initializeApp();
  } catch (e) { /* يعمل التطبيق بدون إعداد Firebase كامل */ }
  await NotificationService.initialize();
  await DatabaseHelper.instance.ensureSeeded();
  runApp(
    MultiProvider(
      providers: [
        ChangeNotifierProvider(create: (_) => AuthProvider()),
        ChangeNotifierProvider(create: (_) => ReportProvider()),
        ChangeNotifierProvider(create: (_) => ResponseProvider()),
        ChangeNotifierProvider(create: (_) => InjuryRecordProvider()),
      ],
      child: const IsnadApp(),
    ),
  );
}
```

- `ensureInitialized` ضروري قبل أي استدعاءات غير متزامنة لعناصر Flutter.
- `initializeDateFormatting('ar')` يحسّن تنسيق التواريخ مع `intl`.
- `ensureSeeded()` يضمن وجود المستخدمين التجريبيين مرة واحدة.
- ترتيب المزوّدات ثابت: المصادقة أولاً منطقياً لأن الشاشات تعتمد على `AuthProvider` لقراءة الدور والهوية.

### ب) إنشاء بلاغ `ReportProvider.createReport`

- يولَّد `report_uuid` عبر `uuid`.
- يُحدَّد `is_synced` من `connectivity_plus` (محاكاة: متصل = يُعتبر «مرسلاً»).
- عند الاتصال قد تُستدعى `NotificationService.showNewReportAlert` لمحاكاة تنبيه مركزي.
- بعد الإدراج تُحدَّث القائمة في الذاكرة (`_reports`) ويُستدعى `notifyListeners()` لتحديث القوائم المفتوحة.

### ج) تحديث الحالة `ResponseProvider`

- `updateReportStatus` يحدّث `reports.status` و`updated_at`.
- `addResponse` يضيف صفاً في `responses` مع `action_type` مطابق لانتقال الحالة (`team_sent` يربط بحالة `responding` في الجدول).
- الدالة الثابتة `statusForAction` هي **نقطة واحدة للحقيقة** لربط نوع الزر بالحالة النصية في `reports`.

### د) المزامنة الوهمية `SyncService.syncPendingReports`

- يقرأ البلاغات حيث `is_synced = 0`.
- يؤخر قليلاً (`Future.delayed`) ثم يضبط `is_synced = 1` ويحدّث `updated_at` — لمحاكاة إتمام رفع دون خادم.

## 11. المصطلحات التقنية

| المصطلح | معنى مبسّط | مثال في إسناد |
|---------|------------|----------------|
| **Widget** | لبنة واجهة في Flutter. | `Scaffold`, `ListView`, `IsnadButton`. |
| **StatefulWidget** | ودجة تحتفظ بحالة تتغير مع الوقت. | شاشات النماذج والخرائط. |
| **Provider** | نمط لتوفير كائنات للأبناء وإعادة البناء عند التغيير. | `context.watch<AuthProvider>()`. |
| **Route** | مسار يحدد الشاشة أو المعاملات. | `/report_detail` مع `arguments: {'reportId': id}`. |
| **SQLite** | قاعدة ملفية محلية. | ملف `isnad.db` في مجلد قواعد التطبيق. |
| **Foreign Key** | ربط صف بجدول آخر. | `responses.report_id` → `reports.id`. |
| **Desugaring (أندرويد)** | دعم ميزات لغة أحدث على إصدارات أقدم من الجهاز. | قد تطلبه إضافات مثل الإشعارات المحلية في إعداد Gradle. |
| **NDK** | أدوات بناء أصلية لأندرويد؛ بعض الإضافات تطلب إصداراً محدداً. | توحيد الإصدار في `build.gradle.kts` يقلل التحذيرات. |

## 12. الأسئلة الشائعة

### عام والتشغيل

**س: التطبيق لا يبنى على أندرويد ويظهر خطأ NDK أو desugaring؟**  
ج: تأكد من إعداد `android/app/build.gradle.kts`: `ndkVersion` موحّد مع ما تطلبه الإضافات، وتفعيل `isCoreLibraryDesugaringEnabled` مع اعتماد `desugar_jdk_libs` إذا ظهرت رسالة AAR metadata. ثم نفّذ `flutter clean` وأعد البناء.

**س: `flutter clean` يفشل أو لا يحذف مجلد `build`؟**  
ج: أغلق Android Studio أو العملية التي تفتح الملفات، أو أعد تشغيل الجهاز، ثم احذف مجلد `build` يدوياً إن لزم، وأعد `flutter pub get`.

**س: المحاكي لا يعطي موقعاً؟**  
ج: من إعدادات المحاكي Extended Controls → Location حدّد نقطة؛ أو استخدم جهازاً حقيقياً مع تفعيل الموقع ومنح الإذن للتطبيق.

**س: الصورة لا تُرفق؟**  
ج: تحقق من أذونات الكاميرا/التخزين/المعرض حسب إصدار أندرويد؛ جرّب المعرض أولاً إذا كانت الكاميرا غير متوفرة على المحاكي.

**س: الخريطة فارغة أو لا تظهر البلاطات؟**  
ج: تحقق من اتصال الإنترنت (البلاطات تُحمّل من الشبكة)؛ تأكد أن البلاغات تحتوي على `location_lat` و`location_lng`.

**س: نسيت كلمة المرور التجريبية؟**  
ج: راجع الجدول في القسم 8؛ أو احذف تطبيق المحاكي/امسح بيانات التطبيق لإعادة إنشاء قاعدة البيانات والبذر.

## 13. الشرح المعماري والتفصيل التقني العميق

### 13.1 طبقات المشروع (Layering)

1. **عرض (UI):** `screens/` + `widgets/` — لا تفتح قاعدة البيانات مباشرة؛ تستدعي `Provider` أو تمرّر أحداثاً.
2. **حالة (State):** `providers/` — يجمع بين قراءة SQLite وتحديث الواجهة عبر `notifyListeners()`.
3. **بيانات (Data):** `models/` + `database_helper` — تمثيل الصفوف ككائنات وتحويلها `Map` ↔ SQLite.
4. **خدمات (Services):** موقع، صور، إشعارات، مزامنة — دوال `static` أو أsingleton بسيطة دون حالة معقدة.

هذا التقسيم يسهّل استبدال طبقة البيانات لاحقاً (مثلاً REST + تخزين مؤقت) مع إبقاء نفس واجهات المزوّدات قدر الإمكان.

### 13.2 آلة حالات البلاغ (State Machine)

الحقل `reports.status` يأخذ قيماً من مجموعة ثابتة تطابق `ReportStatus` في الكود: `open`, `received`, `responding`, `arrived`, `closed`.

| حالة في الجدول | معنى تقريبي | من يقدر يحرّكها |
|----------------|-------------|------------------|
| `open` | بلاغ جديد | القيادة/المستشفى: زر «تم الاستلام» |
| `received` | تم الاستلام | «تم إرسال فريق» |
| `responding` | فريق في الطريق | «تم الوصول» |
| `arrived` | تم الوصول ميدانياً | «إغلاق الحالة» |
| `closed` | مغلق | لا أزرار انتقال لاحقة في المنطق الحالي |

أزرار الواجهة تظهر فقط إذا كانت `status` الحالية تسمح بالخطوة التالية، فيقلّ التناقض بين ما يعرضه المستخدم وما يُخزَّن.

### 13.3 ربط `action_type` في `responses` بحقل `status`

عند الضغط على زر، يُسجَّل صف في `responses` مع `action_type`، ويُحدَّث `reports.status` إلى القيمة المقابلة عبر `ResponseProvider.statusForAction`:

| زر في الواجهة | `action_type` في الجدول | `status` الجديد في `reports` |
|----------------|---------------------------|--------------------------------|
| تم الاستلام | `received` | `received` |
| تم إرسال فريق | `team_sent` | `responding` |
| تم الوصول | `arrived` | `arrived` |
| إغلاق الحالة | `closed` | `closed` |

الفرق بين اسم الزر العربي واسم الحقل مقصود: الجدول يخزّن معرفات إنجليزية قصيرة لتسهيل الاستعلام والتوسعة.

### 13.4 المزوّدات (Providers) ومسؤولياتها

- **`AuthProvider`:** `login` يقرأ من `users` ويطابق كلمة المرور؛ `logout` يمسح مفاتيح الجلسة؛ `checkSavedSession` يعيد تحميل المستخدم من قاعدة البيانات إن وُجد `isnad_user_id` في التخزين المحلي.
- **`ReportProvider`:** يحمّل البلاغات إما لكل المستخدمين (قيادة) أو بفلتر `military_id` (عسكري)؛ `createReport` / `createSOSReport` يضيفان صفاً ويحدّثان القائمة؛ `getReportById` للتفاصيل.
- **`ResponseProvider`:** يحدّث الحالة ويضيف سجل استجابة؛ `getResponsesForReport` يغذّي خط الزمني.
- **`InjuryRecordProvider`:** يحمّل محاضر الجندي مع `JOIN` لإظهار نوع الإصابة من `reports`؛ `createInjuryRecord` بعد حوار المستشفى.

### 13.5 الجلسة (Session) ومفاتيح `SharedPreferences`

بعد تسجيل الدخول الناجح تُحفظ تقريباً: `isnad_user_id`, `isnad_military_id`, `isnad_full_name`, `isnad_role`. عند فتح التطبيق من جديد، `SplashScreen` يستدعي `checkSavedSession` لإعادة بناء `UserModel` من SQLite وليس من نص JSON كامل — هذا يقلّل خطر تعارض البيانات إذا تغيّر المستخدم في قاعدة البيانات يدوياً.

### 13.6 التوجيه (Routing) والمعاملات

المسارات الثابتة مذكورة في `routes` في `app.dart`. مسار **`/report_detail`** وحده يحتاج معرف البلاغ، فيُمرَّر عبر `Navigator.pushNamed(..., arguments: {'reportId': id})` ويُفسَّر في `onGenerateRoute`. أي مسار جديد يحتاج معاملات يمكن إضافته بنفس النمط.

### 13.7 خط الزمني `StatusTimelineWidget`

الواجهة تعرض خمس مراحل ثابتة بالترتيب. الألوان تعبّر عن: مكتمل (أخضر)، حالي (ذهبي)، قادم (رمادي). أوقات المراحل تُستنتج من دمج `created_at` للبلاغ ووقت سجلات `responses` حيث ينطبق، مع عرض نصي عبر `DateFormatter`.

### 13.8 السيناريو الخاص بالمستشفى عند الإغلاق

إذا كان الدور `hospital` والحالة `arrived` والمستخدم يضغط «إغلاق الحالة»، يُعرض حوار لجمع بيانات طبية؛ عند التأكيد يُنشأ `InjuryRecordModel` ثم تُكمل خطوات تحديث الحالة إلى `closed` وإضافة استجابة من نوع `closed`. القيادة قد تغلق من `arrived` دون هذا الحوار — حسب منطق الشاشة.

### 13.9 المزامنة والشبكة في `IsnadApp`

في `initState` لـ `IsnadApp` يُشترَك في `SyncService.connectivityStream`؛ عند `online == true` تُستدعى `syncPendingReports()`. لا يوجد إعادة تحميل تلقائية لقوائم البلاغات بعد المزامنة في هذا الإصدار — يمكن تحسينها لاحقاً باستدعاء `refreshReports` من مزوّد مرتبط.

### 13.10 حدود التصميم (مرجع سريع)

الألوان والخطوط مركزية في `core/constants`؛ القيود مثل عدم استخدام شريط جانبي ملون للبطاقات مذكورة في مواصفات المشروع الأصلية وتنعكس في اختيار الحدود الكاملة والزوايا الموحّدة في الودجات المشتركة.
